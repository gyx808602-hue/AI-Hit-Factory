# 文图生视频优化设计

## 背景

当前文图生视频链路已经具备基础的创建、列表、详情和删除能力，但在用户体验上还有三块明显短板：

1. 创建页只有 `prompt` 单字段输入，缺少“根据主题自动生成文案并允许手动编辑”的双阶段能力。
2. 图片上传仍是手写文件输入，和追爆任务详情页已经收敛好的上传交互不一致。
3. 详情页与列表页缺少针对处理中任务的自动轮询与更清晰的进度反馈。

## 本轮目标

- 在创建页引入“两段式”输入结构：
  - `topic`：主题输入
  - `prompt`：最终提交文案，可由系统生成后继续手动编辑
- 统一图片上传交互，参考追爆任务现有 `antd Upload + beforeUpload + 上传成功回填` 流程。
- 为详情页与列表页补齐处理中任务的自动轮询和进度条展示。

## 非目标

- 不重构文图生视频详情页为追爆任务那种三步工作流。
- 不修改后端现有“创建 / 列表 / 详情 / 删除”接口契约。
- 不把本轮实现扩展成站级通用异步任务框架。

## 关键约束

### 1. 生成文案接口暂未公开

当前在 `用户端1.0.0.md` 中只明确看到文图生视频四个接口：

- `GET /user-api/customer/text-image-video/tasks`
- `POST /user-api/customer/text-image-video/tasks`
- `GET /user-api/customer/text-image-video/tasks/{id}`
- `DELETE /user-api/customer/text-image-video/tasks/{id}`

尚未看到文图生视频专属“生成文案”接口定义。因此本轮需要拆成两层：

- 页面层完整支持 `topic -> 生成文案 -> 手动编辑 -> 创建任务`
- 调用层单独封装“生成文案”能力，优先留出真实接口边界

如果后端接口当前不可用，则页面必须给出明确失败提示，不能假装已经打通远端 AI。

### 2. 追爆任务上传流程只复用交互骨架

本轮参考 `VideoRemixTaskDetailPage.tsx` 的上传做法，但不做大范围共享组件重构。优先目标是把文图生视频创建页的上传体验统一到：

- `Upload` 控件承接选择文件
- `beforeUpload` 拦截默认上传
- 调用上传 API
- 上传成功后回填表单值和预览列表
- 删除预览时同步移除真实提交值

## 方案对比

### 方案 A：最小增强

- 仅在当前创建页增加 `topic` 输入框和一个“AI 生成文案”按钮
- 图片上传改为 `Upload`
- 详情页补基础进度条

优点：

- 改动小，落地快

缺点：

- 列表页状态不会自动刷新
- 生成文案逻辑、表单映射和轮询规则容易继续散落在页面里

### 方案 B：轻量状态增强

- 创建页补齐 `topic + prompt` 两段式结构
- 把文案生成、表单映射和上传回填逻辑收敛到 `features/text-image-video/form.ts`
- 详情页和列表页都补处理中的自动轮询
- API 层预留独立的生成文案封装

优点：

- 结构更清晰，后续接真接口或增加“重新生成”更顺
- 用户在创建页、列表页、详情页看到的状态反馈更一致

缺点：

- 比最小补丁多一些测试和状态维护工作

### 结论

采用方案 B。它能在保持 KISS 的前提下，把本轮最有价值的三件事一次收拢：

- 创建页自动生成文案
- 上传交互统一
- 处理中任务状态反馈增强

## 交互设计

### 创建页

- 保留输入方式切换：`文字输入 / 图片上传 / 图文混合`
- 新增 `topic` 输入框，作为文案生成源
- `prompt` 文本框始终保留，作为最终提交值
- 新增 `AI 生成文案` 按钮：
  - 文字模式：根据 `topic` 生成
  - 图片 / 图文模式：根据 `topic + imageUrls` 生成
- 用户可以继续编辑生成后的 `prompt`
- 点击“开始生成视频”时，仍然走已有创建任务接口

### 上传交互

- 使用 `antd Upload` 作为上传触发器
- 上传成功后立即更新：
  - 页面预览列表
  - 表单里的 `imageUrls`
- 删除预览项时同步删除真实提交值，避免 UI 和 payload 不一致

### 详情页与列表页

- 当任务仍处于处理中状态时，自动轮询最新状态
- 当任务进入成功或失败状态时，自动停止轮询
- 详情页强化进度条显示
- 列表页只做轻量轮询，不增加复杂动作区

## 数据设计

### 表单值

`TextImageVideoFormValues` 扩展为：

- `topic: string`
- `prompt: string`
- `model: string`
- `imageUrls: string[]`

### 生成文案请求边界

前端内部统一使用以下输入结构：

- `topic`
- `imageUrls`
- `inputMode`

这样后续无论是真实后端接口，还是短期前端兜底生成逻辑，都只需要适配这一层输入。

## 测试策略

- 创建页先补失败测试：
  - 两段式字段展示
  - 生成文案按钮行为
  - 上传回填后创建 payload 正确
- 详情页先补失败测试：
  - 处理中自动轮询
  - 完成 / 失败停止轮询
- 列表页先补失败测试：
  - 存在处理中任务时自动刷新

## 风险

- 若后端暂未提供真实生成文案接口，则“AI 自动生成”能力只能先做到前端结构和调用边界完整。
- 当前仓库存在部分历史中文乱码文件，编辑时需尽量避免不相关扩散。
