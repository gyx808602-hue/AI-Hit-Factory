## Context

当前追爆任务相关能力已经在前端形成基本闭环：列表页可以创建和进入详情页，详情页可以保存表单、生成 Prompt、生成视频并刷新任务结果。但页面结构仍然沿用“基础信息 + 内容方向 + 素材 + Prompt 与结果”的堆叠方式，本质上是一个大表单页面，而不是面向创作流程的步骤式工作流。

这次改动的核心不是新增后端能力，而是重构前端交互模型，让页面结构更贴合真实使用顺序：

1. 先创建任务拿到任务 ID
2. 再上传素材并补齐配置
3. 再生成和修改提示词
4. 最后生成视频并查看对比结果

这和后端任务模型是对齐的。后端的 `video-remix-task` 更像一个可反复更新的异步任务容器，前端每一步都是在往这个容器里补数据、触发阶段动作、刷新阶段结果。类比前端概念，它不是“单次提交表单”，而更像“带持久化状态的多阶段 wizard + async job runner”。

当前项目约束也比较明确：

- 技术栈必须继续使用 `React + TypeScript + Ant Design + React Query`
- 请求访问必须继续走 `API Client`，不能在页面里直接写请求
- 服务端状态必须继续由 `React Query` 管理，不能把任务详情塞进全局 store
- 当前仓库已经存在 `src/features/video-remix/form.ts` 和 `src/features/video-remix/status.ts`，适合继续作为本次改造的领域适配层

## Goals / Non-Goals

**Goals:**

- 将追爆任务详情页重构成流程导航式工作流页面
- 保留现有“创建任务入口独立存在”的任务流，避免把创建前状态和创建后状态混在一个路由里
- 将素材上传和配置步骤前置，把“内容方向/复刻方向”移动到素材之后
- 删除不再需要的“视频摘要”输入，并收敛商品图、人物图上传方式为“上传 + 预览 + 删除”
- 将提示词提升为独立步骤，支持查看和编辑
- 在视频生成步骤中明确展示参考视频、生成结果和进度反馈
- 为提示词生成和视频生成增加更细粒度的 loading / progress 表达

**Non-Goals:**

- 不改动后端接口路径和核心返回结构
- 不在本次设计中新增真实的 AI 自动生成后端接口；前端先预留按钮与交互位，具体接入以已有接口能力为准
- 不重构任务列表页为多步骤页面，列表页仍然负责“创建入口 + 进入详情”
- 不把追爆任务并入通用任务中心做统一抽象
- 不引入新的全局状态库、步骤状态框架或表单引擎

## Decisions

### Decision 1：采用“创建入口独立保留，详情页承载后续流程”的分层方案

采用：

- `/viral-remix/tasks` 继续保留创建入口
- `/viral-remix/tasks/:taskId` 重构为步骤式详情流程页

Why：

- 任务创建前后是两个状态域。创建前只有 `name/remark` 等轻量数据；创建后才有任务 ID、任务状态、结果视频和进度。把这两类状态强行塞进一个步骤页，会让路由初始化、刷新恢复和异常处理都更复杂。
- 当前代码已经形成“列表页创建后跳转详情页”的链路，这个边界天然和后端任务主键对齐，重构成本最低。
- 对后端原理做类比，这就像先 `POST` 拿到资源 ID，再进入资源详情页做后续 `PUT/POST action`。前端如果把这个边界打散，后面恢复草稿和处理中状态都会变脆。

备选方案：

- 直接把详情页变成完整 4 步，包含“创建任务”步骤

不选原因：

- 会引入“未创建任务但已进入详情流程”的虚拟状态
- 刷新后必须处理“第 1 步没有 taskId”的边界，交互和实现都更绕

### Decision 2：步骤导航只在详情页内做局部状态，不进入全局 store

采用：

- 当前步骤索引、步骤可达性、步骤内临时提示作为页面级状态
- 任务详情、任务进度、生成结果继续由 React Query 查询结果驱动

Why：

- 步骤切换状态只对当前详情页有意义，不是全局共享状态
- React Query 管理的是服务端任务实体；步骤切换管理的是本地页面组织，两者职责不同
- 把步骤状态提升到全局 store 只会增加复杂度，不会带来复用收益

备选方案：

- 用全局 store 保存“当前任务当前步骤”

不选原因：

- 没有跨页面复用价值
- 容易导致刷新后本地步骤和服务端数据不一致

### Decision 3：继续使用 `features/video-remix` 作为表单与状态适配层，不新增过度抽象组件

采用：

- 在 `src/features/video-remix/form.ts` 扩展字段映射，适配新的步骤布局
- 在 `src/features/video-remix/status.ts` 扩展阶段状态、按钮可用性和进度文案推导
- 如需拆分，仅增加 1~2 个详情页内部使用的步骤区块组件，仍保持在 `pages` 附近或 `features/video-remix/components`

Why：

- 当前需求的复杂点在字段迁移和交互流程，不在复用 UI 组件
- 抽象过度会把一次性页面布局重构，错误包装成“通用步骤组件工程”，反而增加理解成本
- 现在最有复用价值的是“状态判定”和“表单映射”，这两层已经存在，继续强化它们最符合 KISS

备选方案：

- 额外封装通用 `TaskStepFlow`、`AssetUploadPanel`、`PromptEditorPanel` 等大组件

不选原因：

- 当前只有追爆任务使用，抽象收益不足
- UI 结构仍在快速迭代阶段，过早通用化会放大改动成本

### Decision 4：素材步骤收敛为“上传驱动”模型，彻底弱化 URL 文本输入

采用：

- 商品图、人物图只保留上传按钮、缩略图预览、删除动作
- 内部仍然使用 `productImageUrlsText` / `characterImageUrlsText` 作为表单持久化中间层，避免一次性大改接口映射
- UI 层不再向用户暴露 URL 输入框

Why：

- 用户的真实操作目标是上传素材，不是手工维护 URL 列表
- 现有 `form.ts` 和保存接口依赖数组或拼接文本，这属于内部表示，不应该泄露到 UI
- 这相当于后端里的 DTO 和 ViewModel 分离：接口字段可以保留，页面表现不必一一裸露

备选方案：

- 改接口字段结构，直接把前端表单存成文件对象数组

不选原因：

- 会放大改造范围，触及更多现有保存逻辑和测试
- 本次更适合先做“视图层收敛，持久层兼容”

### Decision 5：提示词步骤采用“生成结果 + 可编辑输入”双态合并，而不是只读展示

采用：

- 进入提示词步骤时，优先展示当前已生成提示词
- 同一页面允许直接编辑提示词文本，并作为后续生成视频输入
- 若尚未生成，展示空态和生成按钮；若正在生成，展示 loading 与当前进度文案

Why：

- 用户诉求是“展示生成的提示词，并且可以修改”，这意味着提示词已经从结果数据变成了二次创作输入
- 只读展示无法满足后续微调需求；完全拆成“查看结果页 + 单独编辑页”又会增加路径复杂度

备选方案：

- 继续沿用结果区只读展示
- 新增一个独立弹窗编辑提示词

不选原因：

- 前者不满足需求
- 后者打断步骤连续性

### Decision 6：视频生成步骤增加“参考视频 vs 生成视频”对比区，并把进度反馈集中在步骤区头部

采用：

- 当有参考视频和生成视频时，使用双栏或上下对照区展示
- 将“当前状态、进度条、阶段说明、失败原因、刷新动作”集中放在视频生成步骤顶部
- 生成中不只显示按钮 loading，还显示明确的进度条和阶段文案

Why：

- 用户在视频生成阶段最关心的是“有没有开始跑、跑到哪一步、结果和原视频差别如何”
- 按钮 loading 只能表达“请求已发出”，不能表达“任务仍在处理”
- 这和后端任务监控类似：需要阶段性状态面板，而不只是一个提交按钮

备选方案：

- 沿用原有“结果区底部展示视频 URL/视频播放器”的方式

不选原因：

- 对比感弱
- 进度信息分散，不利于理解当前任务处于哪一阶段

### Decision 7：动态路由层不新增新的业务入口模型，仅复用现有 `routeRegistry`

采用：

- 继续使用当前 `content.viralRemixTasks` 和 `content.viralRemixTaskDetail` 路由键
- 不为步骤再创建子路由，步骤切换只在详情页内部完成

Why：

- 当前步骤并不是权限边界，也不是需要分享链接的独立页面
- 如果把每一步拆成子路由，会引入额外的 URL 同步、刷新恢复、权限映射和菜单高亮复杂度
- 现有动态路由系统负责“页面级访问控制”，不需要下沉到“页面内部步骤级导航”

备选方案：

- 使用 `/viral-remix/tasks/:taskId/materials`、`/prompt`、`/generate` 子路由

不选原因：

- 对当前需求来说过重
- 会增加路由和测试成本，却没有相应收益

## Risks / Trade-offs

- [Risk] 现有保存接口未明确支持“编辑后的 generatedPrompt 反写”为正式输入源  
  → Mitigation：本次设计先在前端明确区分“生成出的提示词”和“当前可编辑提示词”，实现阶段确认最终提交字段映射，必要时通过现有表单字段承接。

- [Risk] “AI 自动生成”按钮若后端暂无独立接口，容易出现前端按钮已出现但无真实动作  
  → Mitigation：在实现阶段先接成显式入口动作和占位交互，若没有后端能力则展示清晰提示，不伪造成功结果。

- [Risk] 商品图/人物图隐藏 URL 输入后，历史测试和表单映射容易失配  
  → Mitigation：保留内部文本映射层，仅替换视图交互；测试同步改为验证预览列表与删除动作。

- [Risk] 步骤化后，表单字段分布更散，保存时机不当会导致用户误以为已持久化  
  → Mitigation：每个步骤保留明确的保存或下一步触发保存逻辑，并在顶部提供统一成功/失败反馈。

- [Risk] 进度条显示依赖后端 `progress`，若后端阶段描述不稳定，前端展示会显得粗糙  
  → Mitigation：前端优先使用真实 `progress/statusLabel`，缺失时再用本地推导文案兜底。

## Migration Plan

1. 保持现有任务列表和路由结构不变，只重构详情页组织方式。
2. 先扩展 `features/video-remix/form.ts` 和 `status.ts`，明确步骤字段归属与阶段状态。
3. 再重构 `VideoRemixTaskDetailPage.tsx`，把单页结构改为步骤导航 + 当前步骤内容区。
4. 同步修改素材上传区交互，移除 URL 输入暴露和视频摘要输入。
5. 补充提示词编辑区和视频对比区。
6. 最后补测试，覆盖：
   - 步骤导航切换
   - 字段顺序与说明文案
   - 上传驱动交互
   - 提示词可编辑
   - 进度和结果对比

回滚策略：

- 本次只涉及前端页面结构和交互层改造，不涉及数据迁移
- 若上线后交互存在问题，可回退 `VideoRemixTaskDetailPage.tsx` 及相关适配层变更，恢复为原有大表单布局

## Open Questions

- “AI 自动生成”按钮最终是各自调用独立接口，还是复用已有提示词生成链路的一部分，需要在实现前再确认一次当前后端能力。
- 编辑后的提示词最终应该落在哪个字段上，是否需要新增单独的可编辑 prompt 字段，还是沿用现有生成结果字段，需要实现前验证现有接口行为。
- 新老视频对比区最终采用左右双栏还是上下布局，可以在实现时根据页面宽度和现有视觉风格做一次轻量取舍，但不影响核心能力定义。
