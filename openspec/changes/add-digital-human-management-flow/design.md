## Context

当前项目已经具备 `React + TypeScript + Ant Design + React Query + request 二次封装` 的基础设施，但 [DigitalHumansPage.tsx](F:\AAA_AI_aisperce\AI-Hit-Factory\src\pages\DigitalHumansPage.tsx) 仍停留在 Figma 演示态：

- 列表来自 `src/features/workspace/mockData.ts`
- 状态是前端本地的“启用/停用”
- 创建弹窗字段是“性别 / 风格 / 音色”这类原型字段
- 卡片动作只在本地内存里切换，没有真实服务端状态

而 `用户端.md` 对“数字人管理”实际提供的是一组数字人形象接口：

- `GET /api/aigc/digital-persons`
- `POST /api/aigc/digital-persons`
- `GET /api/aigc/digital-persons/{id}`
- `DELETE /api/aigc/digital-persons/{id}`
- `GET /api/aigc/digital-persons/{id}/refresh`

这组接口背后表达的不是“开关一个本地卡片”，而是“创建一个异步训练中的数字人资产，再持续查询其状态”。从后端原理看，它更像一个异步资源生命周期管理接口，而不是普通同步表单。你可以把它类比成前端里的“先发起 `mutation` 创建资源，再靠 `query` 观察资源状态变化”的模式；差别只是后端把这个资源真正落库了，所以前端必须有详情页来承接刷新恢复和状态回看。

另外，创建接口本身支持两种互斥输入：

- 方式 A：`multipart/form-data` 直接上传 `file`
- 方式 B：只传远程 `fileUrl`

因此本次设计必须同时处理真实文件提交流和远程 URL 提交流，而不是继续复用当前 mock 弹窗结构。

结合用户补充的参考弹窗，“新建数字人”至少需要明确承载以下 UI 字段与交互：

- `数字人名称`：必填文本输入
- `训练素材`：`本地上传 / 远程 URL` 二选一切换
- `素材 URL`：仅在远程 URL 模式下显示
- `训练类型`：下拉选择，对应 `both / figure / voice`
- `语种`：下拉选择，对应 `cn / en`
- `跳过错误帧`：布尔开关，对应 `errorSkip`
- 底部操作：`取消 / 提交`

也就是说，当前 OpenSpec 不能只停留在“支持文件或 fileUrl”这种抽象描述上，而要把弹窗级字段显式写进需求和任务里，否则实现时仍然可能和参考图偏离。

## Goals / Non-Goals

**Goals:**

- 让数字人管理页真正接入数字人形象库接口，支持服务端分页、搜索、状态过滤、创建、删除和详情跳转。
- 新增数字人详情页，承接训练中、训练成功、训练失败等状态的恢复查看和手动刷新。
- 在 `src/features/digital-human` 内集中数字人状态映射、表单适配和 React Query hooks，减少页面层分散逻辑。
- 保持 `request 基础层 -> API Client -> React Query hooks -> 页面` 的分层，不在页面中直接拼 multipart 请求。
- 在现有动态路由体系内补齐数字人详情页的静态注册与菜单映射边界。
- 让“新建数字人”弹窗在字段结构上与参考弹窗对齐，至少覆盖名称、训练素材切换、素材 URL、本地上传、训练类型、语种、跳过错误帧和取消/提交操作。

**Non-Goals:**

- 不在本次 change 中实现数字人视频任务流，即不接 `digital-person-videos` 这一组接口。
- 不修改后端接口路径、请求参数和状态码定义。
- 不为当前仓库提前抽象“全站异步资源管理框架”，只提取数字人领域最小必要能力。
- 不继续保留 mock 的“启用/停用、性别、风格、音色”作为真实提交字段；这些如果后端没有契约，就不能硬塞进真实请求。

## Decisions

### Decision 1：本次范围只落在 `digital-persons`，不把 `digital-person-videos` 一起打包

采用：

- 本次 change 只覆盖数字人形象库管理
- 数字人视频任务流留给后续独立 change

Why：

- 当前用户请求是“创建数字人管理的任务”，而现有入口页就是 `/digital-humans`，职责更贴近资产管理页。
- 如果同时把“数字人形象”和“数字人视频任务”揉在一次 change 中，会变成“资产管理 + 异步任务中心”的双主题任务，范围和验收都会失焦。
- KISS 原则下，先把形象库真实链路跑通，后面数字人视频任务可以复用状态展示经验，但不应该现在强行捆绑。

备选方案：

- 在一个 change 中同时做 `digital-persons` 和 `digital-person-videos`

不选原因：

- 会明显扩大页面、路由、API、测试和验收范围。
- 会让“数字人管理页”到底是资产页还是任务页的职责边界变得模糊。

### Decision 2：创建流程优先直连 `POST /api/aigc/digital-persons`，不额外拆预上传接口

采用：

- 创建弹窗直接提交后端创建接口
- 文件模式下通过 `FormData` 传 `file`
- 远程模式下只传 `fileUrl`

Why：

- 后端创建接口已经明确支持 `multipart/form-data` 和 `fileUrl` 二选一，这说明“上传 + 创建”本来就是一个完整业务动作。
- 如果前端再额外拆成“先上传素材，再调创建”，会无端引入两次请求、两套状态和更多失败分支，前后端交互成本更高。
- 当前仓库已经有上传能力，但这里没有必要先把数字人训练素材强行纳入统一素材库再创建，这会过度设计。

备选方案：

- 先调用通用上传接口，拿到 URL 后再调用数字人创建

不选原因：

- 增加接口耦合和失败链路长度。
- 后端创建接口已具备直传能力，前端重复拆层没有收益。

### Decision 2.1：创建弹窗字段按“参考 UI + 后端契约交集”落地，而不是继续沿用原型假字段

采用：

- 创建弹窗显式包含：
  - 名称输入框
  - 本地上传 / 远程 URL 切换
  - 远程 URL 输入框
  - 训练类型下拉
  - 语种下拉
  - 跳过错误帧开关
- 不再保留当前 mock 里的性别、风格、音色字段

Why：

- 参考弹窗已经把产品交互粒度给得很具体，如果任务文档不把这些字段写死，后续实现很容易“接口接上了，但弹窗长得不是这一版”。
- 但字段来源不能只看图，还要看后端契约。这里最稳的做法是取“参考 UI 与后端契约交集”，避免出现视觉稿有、接口没有的伪字段。
- 这和后端接口设计里的 DTO 边界是一个道理：页面字段必须最终能稳定映射到请求体，而不是只服务视觉展示。

备选方案：

- 只在 design 里抽象写“支持创建数字人”
- 继续保留 Figma mock 字段，再在实现时临时替换

不选原因：

- 抽象写法无法约束实现细节。
- 保留 mock 字段会直接误导前端和测试用例。

### Decision 3：重构数字人页面语义，从“本地开关卡片”改为“服务端资源生命周期管理”

采用：

- 列表页保留搜索、状态筛选、创建入口、删除入口
- 卡片主动作调整为“查看详情”“刷新状态”“删除”
- 详情页负责展示进度、预览图、预览视频、失败原因和基础元数据

Why：

- 当前 mock 的“启用/停用”状态与后端真实 `status 0~4` 不是一回事，继续沿用会制造假语义。
- 数字人训练天然是异步过程，详情页才是最适合承接进度和失败信息的位置。把所有内容堆在卡片列表里，会让列表页承担过多职责。
- 这和前端把“资源列表页”和“资源详情页”分开是同一个原理；也类似后端把集合资源 `/digital-persons` 和单资源 `/digital-persons/{id}` 分成两个端点。

备选方案：

- 保持单页完成所有查看和刷新动作

不选原因：

- 列表页难以承载完整详情字段。
- 刷新页面后上下文丢失，不利于训练中的资源回看。

### Decision 4：数字人领域能力放在 `src/features/digital-human`，只做最小抽象

采用：

- `status.ts`：状态文案、颜色、结果展示兜底规则
- `form.ts`：创建表单默认值与请求体适配
- `hooks.ts`：列表、详情、创建、删除、刷新等 React Query hooks

Why：

- API 状态映射、表单互斥校验和 multipart 创建动作都属于数字人领域逻辑，不适合留在页面 JSX 中。
- 但当前也只有数字人功能在用这套逻辑，直接提升到 `shared` 仍然过早。
- 这符合现有项目“feature 先落地，稳定后再上升到 shared”的结构规则。

备选方案：

- 全写在页面中
- 直接抽成 `shared/async-resource/*`

不选原因：

- 全写页面会让两个页面重复维护状态判断。
- 直接上升到 `shared` 缺乏足够复用证据，容易形成过度抽象。

### Decision 5：列表分页、搜索和状态筛选全部走服务端参数，不做本地全量过滤

采用：

- 列表查询使用 `pageNum`、`pageSize`、`status`、`keyword`
- 前端只保存当前筛选条件和分页状态
- 数据由 React Query 管理，切换条件时触发新查询

Why：

- `digital-persons` 本质属于企业级资产列表，后续数据量不应假设为很小。
- 列表页应该遵循 `openspec/project.md` 的大数据展示约束，优先使用服务端分页和服务端过滤。
- 这样后续扩展到企业数字人库时，性能和缓存策略也更稳。

备选方案：

- 一次拉全量列表后在前端本地筛选

不选原因：

- 不符合大数据页约束。
- 后续列表扩张后性能和一致性都会出问题。

### Decision 6：状态展示采用“后端 `statusLabel` 优先 + 前端 code 兜底”双轨策略

采用：

- 页面显示优先使用后端返回的 `statusLabel`
- 前端根据 `status/progress/previewVideoUrl/errReason` 提供颜色、按钮可用性和空态兜底

Why：

- 当前接口文档只明确了 `status 0~4` 范围，没有给出每个状态码的完整语义名称。
- 如果前端现在就把每个状态码写死成强语义，后续很容易和后端真实枚举脱节。
- 用 `statusLabel` 承接文案、用前端 helper 承接视觉和交互，是更稳妥的折中。

备选方案：

- 只按前端硬编码状态码展示
- 完全不做前端状态 helper，只原样显示后端文本

不选原因：

- 硬编码容易猜错语义。
- 完全不做 helper 会导致列表和详情页状态风格不一致。

### Decision 7：`errorSkip` 开关的“默认开关状态”先不在任务里写死为视觉稿状态，而以后端默认值为准

采用：

- 任务中明确要求展示 `跳过错误帧` 开关
- 默认值实现阶段以后端契约默认值 `false` 为准，除非产品进一步确认要按参考图默认开启

Why：

- 参考图里开关是开启状态，但接口文档明确写了 `errorSkip` 默认 `false`。
- 这里如果仅凭图片把默认值写成开启，后面就可能出现“UI 看着对，提交语义反了”的问题。
- 视觉默认态和业务默认值冲突时，应该先保留这个冲突并显式记录，而不是偷偷选一个。

## Risks / Trade-offs

- [Risk] 当前接口文档没有完整说明 `status 0~4` 的精确业务语义。
  Mitigation：页面以 `statusLabel` 为主，前端只做保守兜底，不把按钮行为和业务结论过度绑定到猜测状态上。

- [Risk] 创建接口使用 `query + multipart/form-data` 的混合契约，前端封装容易把字段放错位置。
  Mitigation：在 API Client 层集中封装 `FormData` 和 query 参数拼装，并配套接口层测试。

- [Risk] 现有 Figma 原型字段与真实后端字段不一致，直接保留会误导实现。
  Mitigation：本次明确以后端契约为准，移除或降级无契约的 mock 字段，不把伪字段写入真实表单。

- [Risk] 现有 `doc/progress.md` 存在编码异常，直接改写有破坏风险。
  Mitigation：本次先在 `doc/` 下新增 UTF-8 进展补充文档，后续再单独处理全局进展文档编码修复。

## Migration Plan

1. 保持现有 `/digital-humans` 入口不变，先替换其数据来源和卡片动作语义。
2. 新增 `/digital-humans/:humanId` 详情页，并在路由注册表中标记为隐藏菜单页。
3. 新增 `src/api/aigc/digital-persons` 与 `src/features/digital-human`，通过增量方式接入，不影响其他业务页面。
4. 如需回滚，只需移除新加的 API 模块、feature 目录、详情页和新增路由，并将 `DigitalHumansPage` 恢复为 mock 版本；本次不涉及后端迁移。

## Open Questions

- 后端是否有更明确的数字人状态码定义文档；如果有，后续可以把前端状态 helper 从保守兜底升级为精确语义。
- 数字人详情页后续是否需要补“去创建数字人视频”跳转；这取决于下一条 `digital-person-videos` change 的范围设计。
- `跳过错误帧` 开关默认值是否按参考图默认开启，还是按后端契约默认 `false`；当前更建议以后端默认值为准，除非产品明确要求覆盖。
