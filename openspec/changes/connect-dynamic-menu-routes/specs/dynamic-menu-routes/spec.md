## 新增需求

### 需求：动态菜单路由 API 契约
系统应调用 `GET /api/v1/menus/routes` 接口，以获取经过认证的菜单路由元数据。

#### 场景：API 返回路由树
- **当**已认证用户请求当前菜单路由时
- **则**前端接收到一个包含 `code`、`data` 和 `msg` 的结果对象，其中 `data` 包含路由项，这些路由项具有 `path`、`component`、`redirect`、`name`、`meta` 以及可选的 `children` 属性

#### 场景：子路由（children）格式无效
- **当**路由项中的 `children` 属性值不是数组时
- **则**前端必须将其规范化为空数组，而不是导致程序崩溃

### 需求：后端组件白名单映射
系统应通过明确的白名单，将后端的 `component` 标识符映射为前端的静态路由键（static route keys）。

#### 场景：已知的后端组件
- **当**后端返回包含白名单内 `component` 的路由项时
- **则**前端将其映射为相应的 `RouteKey`，并渲染已静态注册的 React 组件

#### 场景：未知的后端组件
- **当**后端返回包含未知 `component` 的路由项时
- **则**前端必须将该路由排除在可渲染路由之外，并避免动态导入后端提供的字符串

### 需求：动态菜单元数据转换
系统应将后端的路由元数据（meta）转换为前端的菜单及路由元数据。 #### 场景：可见菜单项
- **当**后端路由项的 `meta.hidden` 属性缺失或为 `false` 时
- **则**前端在侧边栏菜单中显示该项，并使用 `meta.title` 和 `meta.icon` 进行展示

#### 场景：隐藏路由项
- **当**后端路由项的 `meta.hidden` 属性为 `true` 时
- **则**前端必须将其从侧边栏隐藏；若该项映射到有效的路由键（route key），则仍需保留其路由访问能力

#### 场景：Keep-alive（页面缓存）设置
- **当**后端路由项的 `meta.keepAlive` 属性为 `true` 时
- **则**前端将其映射为路由注册表（route registry）所使用的页面缓存元数据

### 需求：动态路由初始化
系统必须在渲染已认证业务页面之前，根据动态菜单数据初始化受保护路由。

#### 场景：刷新受保护页面
- **当**已认证用户刷新受保护的业务路由页面时
- **则**前端加载 `/api/v1/menus/routes`，重建可访问路由，并渲染目标页面，而不是直接跳转至 404 页面

#### 场景：菜单加载状态
- **当**动态菜单路由尚在加载中时
- **则**前端显示加载中状态或骨架屏（skeleton），而不渲染错误的 403 或 404 页面

#### 场景：菜单加载失败
- **当**因认证过期导致动态菜单路由加载失败时
- **则**前端清除会话状态，并将用户引导至登录或登录过期处理流程

### 需求：重定向处理
系统必须正确处理路由重定向，且不应将外部 URL 视为内部 React 路由进行处理。 #### 场景：内部重定向
- **当**后端路由项包含内部 `redirect` 路径时
- **则**前端在验证目标路径可访问后，**可以**将其用作 React Router 的重定向目标

#### 场景：外部重定向
- **当**后端路由项包含 `http` 或 `https` 重定向时
- **则**前端**不得**将其注册为 React Router 的内部路径；若该路由在菜单中展示，**必须**将其作为外部导航操作进行处理

### 需求：动态路由验证
系统**应当**包含针对路由转换及路由初始化行为的测试。

#### 场景：转换器测试
- **当**执行动态路由转换测试时
- **则**测试用例**应当**涵盖：已知组件映射、未知组件过滤、无效子路由标准化、隐藏菜单处理、keepAlive 映射以及外部重定向处理

#### 场景：构建验证
- **当**实现完成后
- **则** `npm run typecheck`、`npm test` 和 `npm run build` **必须**通过，且不得出现动态路由相关的类型错误
