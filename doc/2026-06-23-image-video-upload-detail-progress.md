### 当前判断
- 这次需求本质上不是单点修补，而是一组前端资源交互收口：
  - 上传入口统一到 `antd Upload`
  - 已上传图片支持删除
  - 详情页支持图片 / 视频回显
  - 详情页支持下载
- 从前端状态设计上看，上传列表不该只维护一个展示数组，而应该维护：
  - `UploadFile[]` 作为 UI 文件列表
  - 业务字段数组（如 `imageUrls`）作为提交载荷
- 知识点拨：
  - 这和前端里“受控组件”和“表单真实值”分层是一个道理。
  - `Upload` 组件像一个带内部事件流的复杂输入框，`fileList` 是展示态，`imageUrls` 才是提交给后端的业务态。
  - 如果两者不拆开，删除、回显、重新编辑时状态会互相污染。

## 第二阶段完成：项目统一上传风格风险评估

### 已完成
- 已补充扫描项目内所有与图片上传相关的主要入口：
  - `src/pages/ImageVideoPage.tsx`
  - `src/pages/ProductVideoPage.tsx`
  - `src/pages/ViralRemixPage.tsx`
  - `src/pages/DigitalHumanVideoTasksPage.tsx`
  - `src/pages/VideoRemixTaskDetailPage.tsx`
- 已确认当前项目的上传交互其实分成 4 种形态：
  - 原生 `input[type=file]` + 本地数组展示
  - `Upload.Dragger` 外壳 + 手动隐藏 input 双通道上传
  - 原生 input 上传后回填 URL 到表单
  - 原生 input 上传后，将多个 URL 以换行文本拼接到 `TextArea`
- 已确认详情回显也存在 3 种形态：
  - 纯文本链接
  - 图片预览 + 链接
  - 视频预览 + 文本地址
- 已确认当前“看起来像统一”，但实际状态模型并不统一：
  - 有的页面维护 `{ name, url }[]`
  - 有的页面维护单个 `url + filename`
  - 有的页面维护 `textarea string`
  - 有的页面直接维护 `File`

### 风险评估
- 低风险：
  - 统一视觉层到 `ant.design Upload` 或 `Upload.Dragger`
  - 给详情页补图片/视频预览和下载按钮
  - 给单图上传页补“删除已上传项”
- 中风险：
  - 把多图上传从自定义数组切到 `UploadFile[]`
  - 把“上传后文本 URL 拼接”的页面改造成真正的文件列表模式
  - 修改现有测试用例，从原生 `change input` 适配到 `Upload` 受控行为
- 高风险：
  - 一次性统一所有上传页面的“状态模型”
  - 直接抽一个全局 Upload 组件替换全部场景
  - 把数字人、图文视频、混剪详情编辑页的上传逻辑一轮全改

### 为什么会有这些风险
- 前端原理上，上传组件统一不只是“换个 UI 外壳”，而是“事件模型和状态模型一起换”。
- `antd Upload` 默认偏向 `fileList` 驱动，而当前仓库很多地方其实是：
  - 上传成功后只关心 URL
  - 或只关心多行文本里的 URL 集合
- 这就像后端把不同业务都强行塞进一个 DTO：
  - 表面复用很多
  - 实际上字段语义不一致
  - 最后删除、回显、编辑、校验都会互相牵连

### 推荐策略
- 推荐做“统一视觉与交互规范”，不要第一轮就“统一全部底层状态模型”。
- 第一轮建议统一这些内容：
  - 全部图片上传入口改为 `ant.design Upload` / `Upload.Dragger`
  - 全部单图/多图入口支持删除已上传项
  - 全部详情页统一支持预览和下载
  - 保留各页面原本的业务值结构，只在页面内部做 `UploadFile[] <-> 业务值` 映射
- 这样做的好处是：
  - 页面体验先统一
  - 接口载荷不动，后端联调风险低
  - 后续如果要抽公共组件，也有稳定样板

### 当前结论
- 你想要“项目统一效果”是可做的，但不建议这一轮就追求“一个组件吃掉全场景”。
- 更稳妥的路线是：
  - 先统一视觉、删除、回显、下载
  - 再视情况第二轮抽公共 upload adapter / upload field

### 下一步
1. 输出 2 套统一方案对比，并给出推荐方案。
2. 如果你认可“先统一体验，再逐步统一状态模型”，我就进入详细设计与实现计划。

## 第三阶段完成：统一改造的风险规避策略确认

### 已完成
- 已确认本轮目标调整为：
  - 继续追求项目统一效果
  - 但实现方式必须以“规避风险”为优先
- 已将统一策略从“全量直接替换”收敛为“分层统一”：
  - 先统一视觉、交互、删除、回显、下载
  - 暂不一轮改掉所有页面的底层业务值结构

### 风险规避策略
- 第一层：统一外观和用户操作路径
  - 上传入口统一到 `ant.design Upload / Upload.Dragger`
  - 上传完成后的删除交互统一
  - 详情页预览和下载入口统一
- 第二层：保留各页面自己的业务值模型
  - 数组 URL 的页面，继续提交数组
  - 单个 URL 的页面，继续提交单值
  - 多行文本 URL 的页面，继续内部维护文本值
  - 只在页面内部增加 `UploadFile[]` 与业务值的映射适配
- 第三层：控制改造范围和顺序
  - 先改图片上传相关页面
  - 再改详情页预览和下载
  - 最后视结果再考虑抽公共适配层

### 当前判断
- 这种做法的本质是“统一体验，不强行统一领域模型”。
- 从工程上更稳，因为：
  - 后端接口载荷基本不变
  - 现有页面逻辑改动边界更清晰
  - 一旦某页出现问题，回退成本更低

1. 输出正式方案，对比“激进统一”和“稳妥统一”两条路线。
2. 默认推荐“稳妥统一”路线，并在你确认后进入测试优先实现。

## 第四阶段完成：统一目标聚焦到用户体验层

### 已完成
- 已确认本轮统一目标不是“所有页面内部实现完全一致”，而是“用户体验效果一致”。
- 已将统一标准收敛到用户可感知的交互层，避免过度改造底层状态导致回归。

### 统一体验基线
- 上传入口样式统一：
  - 都使用 `ant.design Upload / Upload.Dragger`
  - 都有一致的上传提示文案、支持格式说明、上传中反馈
- 上传结果表现统一：
  - 图片都以缩略图卡片或统一文件项展示
  - 单图、多图都支持显式删除
  - 删除后立即同步 UI 与真实业务值
- 详情回显统一：
  - 图片资源优先展示缩略预览
  - 视频资源优先展示可播放预览
  - 文本链接退居为辅助信息
- 下载动作统一：
  - 图片、封面、视频、字幕都提供清晰的下载入口
  - 按钮命名和摆放位置保持一致

### 当前判断
- 这样做可以保证用户感知上的“像同一套系统”，同时不必强行要求每个页面内部都用完全相同的数据结构。
- 这也是当前仓库最合理的 KISS 路线：先统一体验，再决定是否统一底层抽象。

1. 正式给出“激进统一”和“稳妥统一”两套方案对比。
