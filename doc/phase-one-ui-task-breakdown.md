## 1. 当前判断

Figma Make 返回的是一个 React/Vite 原型，而不是单一静态画板。已识别页面与组件包括：

- `App.tsx`：应用壳、顶部栏、页面切换。
- `Sidebar.tsx`：左侧导航、折叠状态、菜单项。
- `Dashboard.tsx`：工作台、统计卡片、快速开始、最近任务。
- `ProductVideo.tsx`：商品视频生成，包含 4 步流程。
- `ImageVideo.tsx`：图文生成视频。
- `ViralRemix.tsx`：爆款视频改编 / 追爆。
- `DigitalHumans.tsx`：数字人管理。
- `TaskRecords.tsx`：任务记录。
- `AssetLibrary.tsx`：素材库。

一期应先完成“根据 Figma 还原平台 UI 页面骨架”，再把业务能力拆成独立任务推进。账号体系、图文生视频、追爆、数字人不应混在同一个实现任务里，否则范围会过大，测试和验收也会失焦。

## 1.1 UI 技术方案确认

一期前端 UI 方案确认使用：

- Ant Design：负责表单、表格、弹窗、抽屉、上传、步骤条、标签页、菜单、通知、空态、骨架屏等复杂交互组件。
- TailwindCSS：负责页面布局、栅格、间距、响应式、内容区排版和少量贴近 Figma 的视觉微调。
- Ant Design `ConfigProvider`：负责暗色主题、品牌色、圆角、字体、组件 token 等统一主题配置。

不采用 shadcn/Radix 作为一期主栈。原因是当前产品是 AI SaaS 工作台，后续账号、企业、积分、任务、素材、审核等页面会大量依赖表单、表格、上传、弹窗和分页。Ant Design 的工程成熟度、中文生态和后台系统适配度更高；shadcn/Radix 更适合高定制前台或组件源码可控诉求强的场景，但会增加复杂表单和表格的维护成本。

## 1.2 窗口兼容性要求

一期 UI 按 PC SaaS 工作台优先设计，但必须保证常见浏览器窗口尺寸可用：

- 重点适配：`1280x720`、`1366x768`、`1440x900`、`1536x864`、`1920x1080`。
- 宽屏优化：`1920px` 及以上内容区需要设置最大宽度或合理栅格，避免卡片被拉得过宽。
- 中小桌面：`1024px - 1279px` 使用紧凑布局，侧边栏可折叠，表格列允许横向滚动。
- 窄屏兜底：小于 `1024px` 时不追求完整移动端体验，但必须保证页面不白屏、不遮挡、不出现不可关闭弹窗；侧边栏可改为折叠或抽屉。
- 高度不足：`720px` 高度下顶部栏、侧边栏和内容区必须各自滚动合理，不能出现主按钮被遮挡。
- 弹窗兼容：Modal/Drawer 在小窗口下必须限制最大高度并允许内部滚动。
- 表格兼容：任务记录、成员类表格、积分流水等宽表必须支持横向滚动或列优先级隐藏。
- 验收方式：实现阶段必须使用浏览器分别检查上述窗口尺寸，并记录发现的问题。

## 2. 推荐任务切分

### 任务 0：Figma UI 页面底座

目标：先把 Figma 中已有的主界面骨架落地，让后续 1-4 任务都能挂到统一应用壳里。

包含范围：

- 创建 Vite + React + TypeScript 前端工程。
- 建立应用壳：侧边栏、顶部栏、内容区、暗色主题变量。
- 还原 Figma 已有页面：工作台、商品视频生成、图文生成视频、爆款视频改编、数字人管理、任务记录、素材库。
- 建立静态路由注册表，后续接动态菜单权限时只映射 `routeKey -> component`。
- 先使用 Mock 数据，保留后续 API Client 接入位置。
- 补充窗口兼容性：覆盖常见桌面窗口、紧凑桌面、窄屏兜底、弹窗高度、表格横向滚动。

复杂度：中等偏高。

是否需要细化：需要。建议拆成 5 个子任务：

1. 工程初始化与主题变量。
2. 应用壳、侧边栏、顶部栏。
3. 工作台、任务记录、素材库公共页面。
4. 三个演示能力页面：图文生视频、追爆、数字人。
5. 商品视频生成页面与生成结果页。

### 任务 1：账号体系

目标：为平台建立登录、身份、企业空间、权限拦截的基础页面和状态契约。

包含范围：

- 登录 / 注册 / 找回密码。
- 微信登录后绑定手机号。
- 个人中心。
- 实名认证。
- 企业认证申请。
- 企业空间首页。
- 成员管理。
- 积分用量。
- 协议签署与权限拦截弹窗。

复杂度：高。

是否需要细化：必须细化。账号体系本质上不是一个页面任务，而是平台身份与权限底座。建议沿用已有 OpenSpec change：`openspec/changes/add-account-system-react-pages/`，并在实现前把任务拆成“认证入口、账户中心、企业空间、权限/协议、Mock API 与状态模型”五组。

知识点拨：账号体系类似前端的全局登录态加路由守卫，但后端必须再次校验。前端隐藏按钮只能改善体验，不能提供真正安全性。

### 任务 2：图文生视频演示

目标：还原 Figma 的 `ImageVideo.tsx` 页面，让用户可以完成图文输入到视频预览的演示闭环。

包含范围：

- 输入方式切换：文字、图片、图文混合。
- 视频主题、文案、图片上传占位。
- 视频风格、输出方式、配音/字幕/BGM 开关。
- 生成中、生成成功、下载、保存到素材库状态。
- 后续接入真实接口前，使用 Mock 任务状态。

复杂度：中等。

是否需要细化：建议细化为两个子任务：

1. 页面 UI 与交互状态。
2. Mock 生成任务与任务记录联动。

### 任务 3：追爆演示

目标：还原 Figma 的 `ViralRemix.tsx` 页面，表达“爆款结构分析 + 原创改编生成”的演示流程。

包含范围：

- 上传爆款视频。
- 原创改编声明。
- AI 分析中与分析结果展示。
- 改编方式：换人物、换商品、仿爆款生成。
- 按模式展示对应配置项。
- 生成中与生成成功状态。

复杂度：中等偏高。

是否需要细化：需要。追爆涉及合规提示和多模式配置，建议拆成：

1. 上传与结构分析演示。
2. 三种改编模式配置。
3. 生成结果与合规提示。

知识点拨：追爆不是“复制爆款”，页面文案和状态必须强调结构学习与原创改编。这个边界未来会影响后端审核、日志和风控。

### 任务 4：数字人演示

目标：还原 Figma 的 `DigitalHumans.tsx` 页面，先完成数字人管理演示闭环。

包含范围：

- 数字人列表、统计、搜索、启用/停用筛选。
- 添加数字人弹窗。
- 模板选择、名称、性别、人设风格、声音。
- 预览、编辑、启用/停用、删除动作的前端状态。
- 后续与账号体系联动：高风险自定义数字人需实名认证与协议签署。

复杂度：中等。

是否需要细化：建议细化为两个子任务：

1. 列表、筛选与卡片操作。
2. 添加数字人弹窗与高风险权限预留。

## 3. 一期推荐实施顺序

1. 任务 0：Figma UI 页面底座。
2. 任务 2：图文生视频演示。
3. 任务 3：追爆演示。
4. 任务 4：数字人演示。
5. 任务 1：账号体系。

原因：Figma 当前已经明确覆盖 2、3、4 以及公共底座，但账号体系页面并未在本次 Figma Make 中出现。先搭 UI 底座可以最快看到完整页面，再用账号体系补齐真实 SaaS 的登录、权限和合规闭环。

如果产品上必须先做账号体系，也可以把任务 1 提前，但视觉还原会依赖我们根据现有暗色设计系统补设计，而不是直接从 Figma 复制页面。

## 4. 待确认问题

- 是否确认使用 Vite + React + TypeScript 作为一期前端工程？当前默认采用该方案。
- UI 组件库已确认使用 Ant Design + TailwindCSS。
- 商品视频生成是否也纳入一期 UI 交付？Figma 已经提供该页面，但用户当前列出的 1-4 中未单独提它。
- 账号体系是否需要先补 Figma 页面，还是先按 PRD/OpenSpec 做业务页面？
