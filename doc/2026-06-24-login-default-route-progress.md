## 如何修改才能正常跳转

### 已完成

- 已基于当前代码确认“正常跳转”的最小控制点只有两个：
  - `src/pages/LoginPage.tsx` 中登录成功后的默认 `redirectPath`
  - `src/app/router/routeRegistry.tsx` 中 `/` 这条首页路由绑定到哪个页面

### 推荐方案

- 方案一：保留 `/` 作为统一首页入口，只把 `/` 改绑到新页面
  - 优点：最稳，后续全站所有“回首页”“默认首页”“直接访问 /”语义一致
  - 缺点：如果历史上已有很多地方默认把 `/` 当工作台，需要一起确认业务含义
- 方案二：保留 `/` 还是工作台，但把登录成功后的默认跳转改成其他页面
  - 优点：改动小，只影响“登录后首次进入”
  - 缺点：系统会出现“登录后去 A，但访问 / 还是工作台”的双入口，不够统一

### 当前建议

- 如果你已经决定不再把“工作台”作为首页，优先推荐方案一：
  - 让 `/` 直接指向你真正想给用户看到的首页，比如 `/tasks` 或 `/assets`
- 如果你只是临时不想让登录后先看到工作台，但又不想动全站首页语义，才用方案二

### 最小改动点

1. 如果要改成“登录后默认进任务记录”，可以只改 `src/pages/LoginPage.tsx`
   - 把默认 `redirectPath` 从 `/` 改成 `/tasks`
2. 如果要让系统真正把“任务记录”当首页，改 `src/app/router/routeRegistry.tsx`
   - 让 `/` 绑定到任务记录页
   - 同时保留 `/tasks` 作为兼容入口或做重定向
3. 如果你还要隐藏“工作台”菜单：
   - 可以保留工作台页面路由但设为 `hideInMenu: true`
   - 或者确认不用后再移除，但前提是登录默认跳转和 `/` 已经先改好

## 是否可以默认跳转到第一个路由

### 已完成

- 已结合当前路由结构确认：可以做“默认跳转到第一个可访问路由”
- 已确认更合适的计算位置不在 `LoginPage.tsx`，而在应用壳层：
  - `src/app/App.tsx`
  - 原因是这里已经拿到了：
    - 登录态
    - 动态菜单路由结果
    - fallback 路由
    - 权限过滤后的可用路由

### 当前判断

- 这个需求能实现，但要定义清楚“第一个路由”到底是什么：
  - 不是 `routeRegistry` 里单纯第一个元素
  - 而应该是“第一个可见、可进入、内部路由”
- 如果未来开启动态菜单：
  - 应优先按 `dynamicRouteState.menuItems` 或最终 `availableRoutes` 计算
  - 这样才会跟当前用户权限保持一致
- 如果继续在 `LoginPage.tsx` 里写死：
  - 登录页并不知道当前用户异步菜单和权限结果
  - 容易出现“登录后跳了一个其实没权限/不存在/被隐藏的页面”

### 推荐实现思路

1. 登录成功后仍先跳到 `/`
2. 在 `App.tsx` 中把 `/` 视为“首页分发入口”
3. 当访问 `/` 时：
   - 若工作台仍存在并想保留，可跳工作台
   - 若工作台已隐藏/移除，则自动跳到第一个可访问内部路由
4. 这个“第一个路由”应过滤掉：
   - `hideInMenu: true`
   - 外链菜单
   - 登录页等公开路由

### 当前建议

- 从架构上看，最稳的是：
  - 保留 `LoginPage` 默认跳 `/`
  - 把“跳去哪个首页”的决策统一收口到 `App.tsx`
- 这样做的原理类似后端网关：
  - 登录页只负责发 token
  - 应用壳层负责按用户当前可访问资源做首页分发
