## 当前判断
- 如果后续只是 URL 前缀或动作后缀变化：
  - 主要改动会落在 `src/api/aigc/video-remix-tasks/index.ts`
  - 配套更新 `src/api/aigc/video-remix-tasks/index.test.ts`
  - 页面层通常可以做到无感切换
- 如果后续变更的是请求参数名、响应字段名、分页字段名：
  - 最优修改位置是 `types.ts` + `form.ts`
  - 原理上这是“适配层隔离”：
    - API 层像前端里的数据源适配器，负责把后端字段翻译成页面能稳定消费的结构
    - 页面层只依赖稳定的领域模型，避免每次接口字段一变就把页面一起拖着改
- 如果新文档同时修改 URL 和字段：
  - 建议继续保持“接口层收口”，不要把后端字段直接散落到页面组件里
  - 这样后续联调成本最低，测试点也最集中

## 推荐优化方向

### 方案一：继续保持当前轻量适配层
- 做法：
  - 仅在 `src/api/aigc/video-remix-tasks` 与 `src/features/video-remix/form.ts` 内部吸收字段变化
  - 页面继续消费 `VideoRemixTask` 与 `VideoRemixTaskFormValues`
- 优点：
  - 改动面最小
  - 最符合当前仓库已有实现习惯
  - 联调回归成本低
- 风险：
  - 如果新文档字段差异很大，单个类型文件会继续膨胀

### 方案二：拆出“原始接口类型”和“页面领域类型”
- 做法：
  - 新增 raw response/request 类型
  - 在 API 层显式做一次字段转换，再输出稳定领域对象
- 优点：
  - 更适合接口频繁变化
  - 后续维护更清晰
- 风险：
  - 当前阶段会比方案一多一层转换代码
  - 对现有轻量结构来说略重

## 当前建议
- 下一次你把新的“用户端 md”给到后，优先采用“方案一”落地。
- 只有当新文档和当前字段差异明显变大，比如：
  - 分页结构改了
  - `form` 嵌套被打平或反过来嵌套
  - 动作接口返回体和详情返回体不一致
  - 同一字段在不同接口里命名不统一
- 再升级到“方案二”。
